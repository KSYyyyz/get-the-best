# V2-1.0 第一局目标链路与基础动画规格

日期：2026-05-22

## 1. 阶段定位

V2-1.0 是从 V2-0 空间沙盒进入 V2-1 第一局闭环的第一步。主线目标仍是“研发区 -> MVP -> 销售区 -> 首批用户/收入 -> 月报 -> 阶段复盘”的 20-30 分钟第一局，但本阶段不一次性做完整局内经济深度，而是先建立玩家能看懂的目标链路和员工行动表现。

本阶段采用“目标链路先行，基础动画随用随接”的策略。也就是说，动画不作为独立美术大工程提前吞掉主线节奏，但凡第一局闭环中需要员工行动可读性的位置，必须同步接入基础动画状态。

## 2. 目标

- 在 HUD 或目标提示中展示当前第一局阶段、下一目标和核心进度，例如“研发 MVP”“准备销售”“获得首批用户”“查看月报”。
- 员工自主行动必须服务第一局目标链路：工程师优先研发，策划/设计可参与白板，运营/市场可进入销售或服务器相关设施。
- Godot 前端只消费 C# Core 的 tick、intent、company delta、presentation event，不复制 MVP、用户、收入、阶段推进等经营规则。
- 当员工因为 Core 意图移动到设施时，画面上至少表现出基础状态变化：行走中、坐下/工作中、站立使用设施、离开设施。
- 现有程序化动作可以保留，但必须封装成可替换的动画状态层，后续替换为真实骨骼动画时不改经营桥接逻辑。

## 3. 非目标

- 不在 V2-1.0 内完成完整美术资产替换。
- 不在 Godot UI 中硬编码经营阶段结算。
- 不做复杂 GOAP、排班、会议、培训、厕所、吃饭等完整员工行为系统。
- 不做完整月报界面，只做能进入月报/复盘链路的最小入口和可验证状态。
- 不把 HUD 扩成旧式大面板；办公室空间仍是主画面。

## 4. 推荐实现切分

### 4.1 第一局目标状态层

新增或扩展一个表现层状态模型，用来把 Core 输出转换为前端可读目标：

- 当前阶段：原型 / MVP / 销售准备 / 首批用户 / 月报。
- 核心进度：MVP 进度、现金变化、用户变化、MRR 或最近指标。
- 下一目标文案：玩家现在应该观察或操作什么。

这个状态层只能从 `V2CoreBridge` 或 Core 快照派生，不直接计算经营规则。

### 4.2 目标 HUD 与空间反馈

目标提示应轻量显示在现有顶部 HUD 或目标区，不遮挡办公室主体。优先展示三类信息：

- 当前目标，例如“推进 MVP 到 100”。
- 最新 Core 事件，例如“工程师前往办公桌：MVP 未完成，办公桌可用”。
- 经营变化，例如“现金 -25，MVP +10”。

空间中员工、设施、房间状态要与目标 HUD 对齐：如果 HUD 说员工正在研发，画面上员工必须在办公桌或白板等设施旁执行对应状态。

### 4.3 基础动画状态层

新增或整理员工动画状态，不直接绑定最终素材格式。建议先定义以下状态：

- `Idle`：站立待机。
- `Walking`：移动路径中，使用平滑插值和简化步行动作。
- `SittingDown`：到达办公桌座位后的短过渡，可先用缩放、朝向和高度变化模拟。
- `WorkingAtDesk`：坐在椅子侧面向电脑，播放打字/手部摆动。
- `UsingStandingFacility`：站在白板、服务器等设施前，面向设施并播放轻微交互动作。
- `LeavingFacility`：结束设施使用后回到站立状态。

现有 `TypingHands` 和程序化工作姿态保留为短期实现，但要挂到动画状态层，而不是散落在员工行为代码里。

### 4.4 Core 意图到表现的映射

`EmployeeActionCandidateKind` 到表现层的默认映射：

- `WorkAtDesk` -> 移动到办公桌座位 -> `SittingDown` -> `WorkingAtDesk`。
- `UseWhiteboard` -> 移动到白板前站位 -> `UsingStandingFacility`。
- `MaintainServer` -> 移动到服务器前站位 -> `UsingStandingFacility`。
- `Rest` -> 移动到休息设施 -> 先用 `UsingStandingFacility` 或 `Idle` 表示，等休息设施资产确定后再扩展。
- `Idle` -> `Idle`。

如果 Core 没有给出 `SourceAction`，前端可以退化为现有“前往设施/正在使用”，但不补算候选理由。

## 5. 数据流

1. `OfficeSimulationEngine.Advance(snapshot)` 产出 `SimulationTickResult`。
2. `V2CoreBridge` 转换 `NextSnapshot`、`CompanyDelta`、`Intents`、`PresentationEvents`。
3. `EmployeeAutonomyController` 消费员工 intent，决定移动目标和表现状态。
4. `Employee3DRenderer` 根据表现状态播放平滑移动、工作姿态和设施使用动作。
5. `BusinessFeedbackHudController` 或新增目标 HUD 控制器展示阶段、下一目标和最近事件。

任何经营数字的变化都来自 Core，Godot 只做展示和动画播放。

## 6. 验收标准

- 启动主场景后，玩家能看到当前第一局目标和最近经营反馈。
- 至少一个员工能根据 Core A0.27 意图自主前往办公桌，并进入可见“坐在椅子侧工作”的状态。
- 至少一种站立设施行为可见，例如白板或服务器前的站立使用状态。
- 员工移动期间不再只是一格一格瞬移，必须保持平滑移动。
- 员工普通移动、设施使用、工作完成后，状态标签不能残留“移动中”。
- 本阶段如果没有真实骨骼动画，也必须保持动画状态接口清晰，后续可替换为 GLB/AnimationPlayer。
- MCP 实机验证必须截图并检查 `get_errors` 为 0。
- `python -m ruff check .`、`python -m black --check --line-length 100 --target-version py311 .`、`python -m isort --check-only --profile black --line-length 100 .`、`pytest tests/ -q`、Core 测试、Godot C# build、文档检查必须通过。

## 7. 实施顺序建议

1. 先加测试保护：目标链路文档/脚手架测试、Core intent 到动画状态映射测试、移动状态清理测试。
2. 建立员工动画状态枚举和渲染入口，不立刻换素材。
3. 把现有平滑移动、工作姿态、`TypingHands` 接到动画状态层。
4. 增加第一局目标 HUD 的最小展示。
5. 用 MCP 实机验证员工从空闲到前往办公桌再到工作中的完整链路。

## 8. 风险和约束

- 如果直接导入复杂动画资产，可能被骨架、缩放、导入设置拖慢主线。因此 V2-1.0 只要求“动画状态可替换”和“基础动作可读”。
- 如果目标 HUD 做得太重，会重新滑向旧面板游戏。因此本阶段 HUD 只显示目标、指标和最近反馈，不做大面板管理。
- 如果前端为了让目标链路更明显而硬编码 MVP 或用户变化，会破坏 C# Core 边界。所有经营推进必须从 Core 结果读取。
