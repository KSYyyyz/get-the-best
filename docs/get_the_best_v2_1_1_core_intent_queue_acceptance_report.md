# V2-1.1 Core 意图队列与非办公桌设施表现验收报告

日期：2026-05-22

## 本轮目标

V2-1.0 已经建立第一局目标 HUD 与基础动画状态层，但普通运行时更容易观察到办公桌工作，白板、服务器等非办公桌设施意图不够稳定可见。本轮目标是在不复制经营规则的前提下，让 Godot 表现层完整消费同一批 C# Core 员工意图，避免第一个员工到位后其他员工被 Core 生命周期提前推进成“使用中”，从而让第一局自然出现白板等非办公桌设施互动。

## 已完成内容

1. Core 意图队列
   - `EmployeeAutonomyController` 新增 `_pendingCoreIntentResult`。
   - 当 C# Core 一次 tick 规划出多个 `MoveToFacility` 意图时，Godot 会缓存本批结果。
   - 每个员工完成设施移动后，优先继续消费同一批 Core 意图，再等待下一次 Core tick。

2. 避免表现层跳步
   - 员工到达设施后不再立刻调用 `AdvanceAndApplyCoreSimulation()`。
   - 这样其他尚未视觉移动的员工不会被提前映射成“正在使用设施”。
   - Core 仍然是规则来源，Godot 只负责把 Core 的意图按可见动作顺序播放出来。

3. 非办公桌设施行动标签
   - 办公桌：`工作中`。
   - 产品白板：`讨论方案`。
   - 服务器机柜：`维护服务器`。
   - 这些标签只描述表现状态，不在 Godot 中计算经营收益。

## 验收证据

- 脚手架测试新增 V2-1.1 覆盖：
  - Core 意图缓存存在。
  - 缓存意图会被连续消费。
  - 员工到达后不立即推进 Core tick。
  - 白板和服务器使用标签存在。
- Godot C# 构建通过。
- MCP 实机运行确认：
  - `Employee_2` 移动到白板交互位。
  - `Employee_2/ActivityBadge` 文本为 `讨论方案`。
  - `Employee_2` 没有 `TypingHands`，未被误表现为办公桌工作。
  - `get_errors` 为 0。

## 后续限制

当前仍是第一局表现节奏基线。服务器维护能显示标签和站立使用状态，但普通第一局初始目标仍以 MVP/白板为主，服务器维护需要后续阶段通过 Core 的产品稳定性、上线后用户压力或运营岗位节奏自然触发。
