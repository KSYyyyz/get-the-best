# V2-0.29 员工吸附工位与工作姿态验收报告

日期：2026-05-22

## 本轮目标

本轮只处理 Godot 桌面前端表现层，不在 Godot 中复制经营规则。目标是让“员工与设施交互”从文字提示推进到可见空间行为：

- 员工点击拿起后，移动到办公桌或座位区域，再次点击可吸附到工位。
- 员工进入工位后显示“工作中”，并保持面向电脑的工作姿态。
- 手动拖拽移动完成后，不再残留“移动中”状态。
- 工作表现只负责前端可读性，产出、现金、用户、MRR 等经营结果继续由 C# Core 驱动。

## 已完成内容

1. 员工拖拽增加设施目标识别。
   - 鼠标移到办公桌设施格或对应座位格时，会优先识别为工位交互目标。
   - 目标合法时提示“开始工作”，员工预览落点吸附到座位格。

2. 员工放到办公桌后进入手动工作表现。
   - 员工位置吸附到桌椅座位点，而不是停在设施中心。
   - 员工朝向电脑屏幕，缩放略降并前倾，表现为坐在椅子上工作。
   - 活动标签从旧的“正在使用”改为更明确的“工作中”。

3. 清理拖拽状态残留。
   - 普通手动移动成功、原位放下、非法放回都会通知自主行动表现层清理旧标签。
   - 避免拖拽结束后继续显示“移动中”。

4. 保持规则边界。
   - 本轮没有在 Godot 中增加产能、收入、疲劳等经营算法。
   - 手动吸附工位只更新前端位置和表现状态；后续经营结算仍等待 C# Core 的生命周期与设施占用结果。

## 已知限制

当前员工模型仍是低多边形占位 GLB，没有可直接播放的坐姿、打字、走路骨骼动画。因此本轮实现的是“程序化工作姿态”：

- 通过座位锚点、朝向、缩放、前倾和轻微打字摆动模拟工作中。
- 这比原先上下跳动更接近工位工作，但还不是最终真实骨骼动画。
- 下一步需要按 `docs/get_the_best_v2_animation_asset_plan.md` 引入统一角色骨架和动作库。

## 验收要点

- 员工拖到办公桌或椅子位置后，点击放下可以进入“工作中”。
- 员工工作时应位于椅子侧，面向电脑，而不是站在桌面中心。
- 员工普通拖拽结束后不应残留“移动中”。
- `get_errors` 必须无错误。
